# التغليف والإصدارات والنشر داخل الفريق

اللحظة الفاصلة ليست عندما تعمل الإضافة على جهازك، بل عندما يستطيع شخص آخر أن:

- يجدها
- يفهمها
- يثبتها
- ويحدّثها

بدون جلسة دعم طويلة منك.

## لماذا `version` مهم حتى في الاختبارات؟

لأن فريقك سيحتاج مع الوقت إلى:

- التمييز بين نسخة مستقرة وتجريبية
- معرفة ما إذا تغير behavior
- ربط المشاكل بإصدار محدد

## متى أستخدم Git-backed marketplace؟

استخدمه عندما تريد:

- استضافة plugins في repo مستقل
- مشاركة marketplace عبر `owner/repo`
- تثبيت plugins من GitHub مع `ref` أو `sha`

## ماذا عن النشر العام؟

بحسب الوثائق الحالية، **إضافة plugins إلى الدليل الرسمي العام قادمة لاحقًا**، وكذلك self-serve publishing. هذا يعني أن المسار العملي اليوم غالبًا هو:

- marketplace خاص بالفريق
- أو marketplace خاص بالمستودع
- أو repo Git-backed داخلي

{% accordion title="استنتاج مهم" open=true %}
إذا كان هدفك الحالي هو تمكين فريقك، فلا تنتظر الدليل العام الرسمي. ابنِ marketplace داخليًا الآن، وصمّم الإضافة كما لو أنها ستُنشر لاحقًا.
{% /accordion %}

## Checklist قبل طرح الإضافة على الفريق

{% steps %}
{% step title="ثبّت اسمًا نهائيًا" hint="لا تكثر من تغييره" %}
اسم plugin هو هوية الحزمة وnamespace داخلي، لذلك تغييره لاحقًا مكلف.
{% /step %}
{% step title="أضف metadata المكتملة" %}
أكمل `author` و`homepage` و`repository` و`license` والروابط القانونية إذا كانت الإضافة ستغادر نطاقك الشخصي.
{% /step %}
{% step title="راجع prompts الافتراضية" %}
`defaultPrompt` ليست زينة؛ هي نقطة البدء التي سيراها المستخدم.
{% /step %}
{% step title="اختبر على جهاز آخر أو مستخدم آخر" %}
إن عملت الإضافة فقط لأنك تعرف كل أسرارها، فهي ليست جاهزة للتوزيع بعد.
{% /step %}
{% /steps %}

- اختبر الإضافة مع مستخدم آخر قبل اعتمادها

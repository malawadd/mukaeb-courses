# المصادقة والسياسات والصلاحيات

بمجرد أن يصبح Plugin قادرًا على الوصول إلى apps أو MCP أو خدمات خارجية، تدخل منطقة أكثر حساسية: **المصادقة والصلاحيات ومشاركة البيانات**.

## سياسات التثبيت في marketplace

كل plugin entry ينبغي أن تحتوي على:

- `policy.installation`
- `policy.authentication`
- `category`

والقيم الشائعة لـ `policy.authentication` هي:

- `ON_INSTALL`
- `ON_USE`

## ماذا يحدث بعد التثبيت؟

وثائق `Plugins` تشرح أن:

- تثبيت الإضافة يجعل workflows متاحة داخل Codex
- لكن **approval settings** الخاصة بك تبقى مطبقة
- وأي خدمة خارجية مرتبطة بالإضافة تبقى خاضعة لسياساتها الخاصة في الخصوصية والمشاركة والمصادقة

{% callout title="لا تخلط بين التثبيت والثقة المطلقة" tone="warning" %}
تثبيت Plugin لا يتجاوز قواعد الأمان الأخرى في Codex. إذا كانت هناك أوامر تحتاج صلاحيات أعلى أو سياسات موافقات، فستبقى هذه القيود سارية.
{% /callout %}

## أين تدخل Rules في الصورة؟

وثائق `Rules` توضّح أن Codex يحمّل قواعد من طبقات متعددة، بما فيها user layer وteam config وproject layers. وهذا مهم جدًا لأن الإضافة قد توفر workflow قويًا، لكن rules هي التي تحدد:

- ما الذي يسمح به تلقائيًا
- ما الذي يحتاج موافقة
- ما الذي يجب منعه أصلًا

## كيف تصمم Plugin آمنًا؟

{% steps %}
{% step title="قلل السطح الهجومي" hint="ابدأ ضيقًا" %}
لا تضم داخل الإضافة أكثر من app أو server مما تحتاجه المشكلة فعلًا.
{% /step %}
{% step title="صف البيانات التي تمر" %}
إذا كانت skill ترسل محتوى إلى app أو service خارجية، اجعل هذا واضحًا في الوصف الداخلي وفي الروابط القانونية داخل `interface`.
{% /step %}
{% step title="افصل بين setup والاستخدام" %}
ضع `ON_INSTALL` أو `ON_USE` بقرار واعٍ، لا بالقيمة الافتراضية فقط.
{% /step %}
{% step title="اختبر السلوك تحت قواعد حقيقية" %}
جرّب الإضافة داخل بيئة تملك rules وموافقات قريبة من بيئة فريقك الفعلية.
{% /step %}
{% /steps %}

{% checkpoint title="مبدأ الأمان العملي" %}
- اختر `ON_INSTALL` أو `ON_USE` بناءً على تجربة الاستخدام الحقيقية
- لا تفترض أن التثبيت يعني السماح المطلق
- صمّم الإضافة بحيث تكون واضحة قانونيًا وتشغيليًا
- اختبرها تحت rules واقعية لا في بيئة مثالية فقط
{% /checkpoint %}

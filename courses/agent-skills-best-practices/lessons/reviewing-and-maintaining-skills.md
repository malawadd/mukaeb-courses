# مراجعة المهارة وتطويرها

النسخة الأولى من أي Skill ليست النهاية. المستودعات القوية تتعامل مع المهارة كشيء **يُختبر ويُقارن ويُراجع مع الوقت**، لا كملف يُكتب مرة واحدة ثم يُنسى.

{% callout title="المراجعة الحقيقية تبدأ بعد أول استخدام" tone="warning" %}
السؤال ليس: هل يبدو `SKILL.md` جيدًا؟ بل: هل تفعّلت المهارة عندما كان يجب؟ وهل نفّذت workflow الصحيح؟ وهل أدّت أفضل من baseline أو على الأقل أوضحت السلوك أكثر؟
{% /callout %}

## ماذا نراجع تحديدًا؟

- دقة الـ triggering
- وضوح boundaries
- جودة الـ workflow والـ decision tree
- الحاجة إلى نقل أجزاء إلى `references/` أو `scripts/`
- صلاحية الأمثلة والروابط والمسارات

## لماذا نحتاج evals؟

لأن الانطباع وحده لا يكفي. إذا كانت المهارة:

- تولّد ملفات
- تتعامل مع صيغ أو بنى ثابتة
- تعتمد على اكتشاف trigger صحيح
- تهدف إلى تقليل الوقت أو الأخطاء

فالأفضل أن نمتلك prompts اختبارية وحالات مقارنة واضحة.

## دورة تطوير عملية

{% steps %}
{% step title="اكتب 2 إلى 5 prompts حقيقية" hint="ليس prompts مثالية" %}
اختر طلبات تشبه ما سيقوله المستخدم فعلًا، لا جملًا مدرسية مختصرة.
{% /step %}
{% step title="جرّب with-skill وبدونها أو بالإصدار السابق" %}
هذا يكشف هل تضيف المهارة قيمة حقيقية أم مجرد ضجيج.
{% /step %}
{% step title="حلّل أين فشلت" %}
هل المشكلة في الوصف؟ أم في القرار؟ أم في غياب مرجع أو سكربت؟
{% /step %}
{% step title="عدّل بأقل تغيير فعّال" %}
ضيّق trigger، حسّن boundaries، أو أضف ملفًا مساعدًا بدل زيادة الحشو داخل الملف الرئيسي.
{% /step %}
{% step title="أعد الاختبار" %}
التطوير الجيد دوري، لا لحظي.
{% /step %}
{% /steps %}

## إشارات أن المهارة تحتاج refactor

{% tabs %}
{% tab label="Trigger problem" %}
لا تتفعّل إلا إذا ذكر المستخدم اسمها حرفيًا، أو تتفعّل في مهام مجاورة لا تخصها.
{% /tab %}
{% tab label="Body problem" %}
الوكيل يقرأ المهارة لكنه يظل يتصرف بعشوائية لأن الخطوات أو الحدود غير حاسمة.
{% /tab %}
{% tab label="Structure problem" %}
كل استدعاء ينتهي بكتابة سكربت جديد من الصفر أو قراءة ملفات طويلة لا لزوم لها.
{% /tab %}
{% /tabs %}

## Checklist صيانة

- هل description ما زال يعكس الاستخدام الفعلي؟
- هل أمثلة التفعيل تعكس لغة المستخدمين الحقيقية؟
- هل توجد تعليمات باتت منسوخة من docs قديمة؟
- هل script متكرر يجب أن يصبح ملفًا دائمًا؟
- هل المهارة متضخمة ويجب تقسيمها أو تضييق نطاقها؟

## لا تخلط بين "تكبير المهارة" و"تحسين المهارة"

أحيانًا أفضل إصلاح ليس إضافة 100 سطر، بل:

- حذف تكرار
- نقل مرجع طويل إلى ملف جانبي
- تضييق trigger
- توضيح حالة عدم الاستخدام

# كيف تكتب SKILL.md قويًا؟

ملف `SKILL.md` الجيد ليس مقالًا طويلًا، بل مستند تشغيل عملي. كل قسم فيه يجب أن يقلل الغموض ويزيد احتمال أن يتصرف الوكيل بشكل صحيح من أول محاولة.

{% callout title="ابدأ من سلوك الوكيل لا من شكل الملف" tone="note" %}
قبل كتابة أي سطر، اسأل: ما القرار الذي يجب أن يتخذه الوكيل؟ ما الإشارات التي تخبره أن هذه المهارة هي المناسبة؟ وما الأخطاء التي أريد منعه منها؟
{% /callout %}

## الأقسام التي تحتاجها غالبًا

1. ما الهدف من المهارة.
2. متى يجب استخدامها.
3. متى يجب عدم استخدامها.
4. الخطوات العملية أو workflow.
5. القيود أو الأدوات المسموحة والممنوعة.
6. أمثلة على مدخلات أو مخرجات جيدة.

## قالب عملي مقترح

```md
# Skill Name

## Purpose
اشرح الهدف العملي للمهارة في سطرين أو ثلاثة.

## Use this skill when
- حالة تفعيل واضحة
- حالة ثانية يمكن اكتشافها من الطلب

## Do not use this skill when
- حالة مشابهة لكنها خارج النطاق

## Workflow
1. اجمع الإشارات الحاسمة أولًا.
2. اقرأ الملفات أو الرموز المطلوبة.
3. نفّذ التغيير أو التحليل على أضيق نطاق ممكن.
4. تحقق من النتيجة.

## Output expectations
- شكل النتيجة
- ما الذي يجب أن يتضمنه الرد النهائي
```

## ما الذي يجعل المهارة قابلة للتفعيل فعلًا؟

{% accordion title="سوء شائع: مهارة واسعة أكثر من اللازم" open=true %}
إذا كتبت أن المهارة تُستخدم عندما يطلب المستخدم "المساعدة في البرمجة"، فأنت لم تساعد الوكيل. هذا الوصف واسع جدًا. الوكيل يحتاج مؤشرات أقرب إلى الواقع، مثل أسماء ملفات، أنواع مهام، أو أنماط طلبات واضحة.
{% /accordion %}

## أفضل الممارسات

{% steps %}
{% step title="اجعل trigger محددًا" hint="أهم خطوة" %}
اكتب الحالات التي تستدعي المهارة بلغة يمكن ملاحظتها: نوع ملف، صيغة طلب، نظام معين، أو مشكلة متكررة.
{% /step %}
{% step title="اكتب حدود النطاق بوضوح" %}
وجود قسم "لا تستخدم هذه المهارة عندما..." يقلل التفعيل الخاطئ ويمنع التضارب مع مهارات أخرى.
{% /step %}
{% step title="اربطها بسير عمل يمكن اختباره" %}
إذا لم تستطع تحويل المهارة إلى خطوات قابلة للتحقق، فهي لا تزال مبهمة أكثر من اللازم.
{% /step %}
{% step title="أضف مثالًا واحدًا على الأقل" %}
الأمثلة تقلل التأويل وتوضح مستوى التفاصيل المتوقع من الوكيل.
{% /step %}
{% /steps %}

{% checkpoint title="خلاصة التصميم الجيد" %}
- اكتب trigger واضحًا وقابلًا للملاحظة.
- ضع حدودًا تمنع الاستخدام الخاطئ.
- حوّل المهارة إلى workflow يمكن اختباره.
- استخدم مثالًا أو اثنين لتقليل الغموض.
{% /checkpoint %}

# كتابة SKILL.md قوي

بعد أن فهمنا منطق التفعيل ووصف المهارة، نصل الآن إلى قلب العمل: **كيف نكتب body المهارة نفسها؟**

الملف الجيد ليس مقالًا طويلًا ولا قائمة أمنيات. هو مستند تشغيلي يساعد الوكيل على:

- فهم الهدف
- معرفة متى يتابع ومتى يتوقف
- اختيار الخطوات الصحيحة
- إنتاج نتيجة يمكن تقييمها

## أقسام أساسية ننصح بها

1. عنوان واضح للمهارة.
2. Purpose أو الهدف العملي.
3. Inputs أو الافتراضات المتوقعة.
4. Workflow أو الخطوات الرئيسية.
5. Boundaries أو حالات عدم الاستخدام.
6. Output expectations أو شكل النتيجة.
7. References/Scripts عند الحاجة.

## قالب عملي

```md
# Skill Name

## Purpose
اشرح ما الذي تمكّن هذه المهارة الوكيل من فعله.

## Inputs
- الملفات أو السياق أو الافتراضات المتوقعة

## Workflow
1. اجمع الحد الأدنى من السياق.
2. اختر المسار الصحيح.
3. نفّذ الخطوة أو استدعِ السكربت المناسب.
4. تحقّق من النتيجة.

## Boundaries
- لا تستخدم هذه المهارة عندما...

## Output
- ما الذي يجب أن يسلّمه الوكيل أو يذكره في النهاية
```

## ما الذي يجعل `SKILL.md` قابلًا للتنفيذ؟

{% accordion title="المشكلة الشائعة" open=true %}
كثير من الملفات تقول أشياء صحيحة لكنها غير قابلة للتنفيذ، مثل: "اتبع أفضل الممارسات" أو "تعامل بحذر". هذه ليست خطوات، بل نوايا.
{% /accordion %}

## حوّل النوايا إلى تعليمات قابلة للملاحظة

بدل:

- "افهم المشكلة جيدًا"

اكتب:

- "اقرأ الملف المصدر أولًا"
- "تحقق من وجود المتغير الفلاني أو الخدمة الفلانية"
- "إن كانت الحالة static، استخدم المسار A. وإن كانت dynamic، استخدم المسار B"

## قسم Boundaries ليس اختياريًا

أفضل Skills لا تقول فقط "استخدمني عندما..." بل تقول أيضًا:

- لا تستخدم هذه المهارة للمهام العامة
- لا تعدّل أنواع ملفات معينة
- لا تفتح ملفات ضخمة ما لم تكن ضرورية
- لا تفسّر الطلب على أنه تنفيذ إذا طلب المستخدم guidance فقط

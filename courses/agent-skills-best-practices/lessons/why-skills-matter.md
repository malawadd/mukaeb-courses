# لماذا Skills أصلًا؟

ليس كل سلوك متكرر يحتاج إلى Skill. لكن عندما تبدأ بملاحظة أن الوكيل ينجح أحيانًا ويفشل أحيانًا أخرى في نفس نوع المهمة، فهذه غالبًا إشارة أن المعرفة موجودة في رأسك أنت، لا في النظام.

{% callout title="الفكرة الأساسية" tone="tip" %}
الـ Skill مناسبة عندما نريد تحويل خبرة قابلة للتكرار إلى workflow يمكن لوكيل آخر اكتشافه وتنفيذه بثبات.
{% /callout %}

## متى تكفي التعليمات العامة؟

- عندما يكون المطلوب مجرد تفضيل أسلوبي مثل الاختصار أو نبرة الرد
- عندما تنطبق القاعدة على كل المهام تقريبًا
- عندما لا توجد خطوات تنفيذية أو أدوات خاصة

## متى تصبح Skill هي الخيار الصحيح؟

- عندما توجد مهمة متكررة ذات حدود واضحة
- عندما يحتاج الوكيل إلى معرفة **متى** يستخدم هذا السلوك
- عندما توجد أخطاء شائعة تريد منعها مسبقًا
- عندما تحتاج المهمة إلى أدوات أو مراجع أو مخرجات خاصة

## الفرق بين Preference وRule وSkill

{% tabs %}
{% tab label="Preference" %}
تفضيل عام مثل: "كن مختصرًا" أو "استخدم العربية".
{% /tab %}
{% tab label="Rule" %}
قاعدة أوسع تحكم العمل كله، مثل عدم تعديل ملفات لا تخص المهمة.
{% /tab %}
{% tab label="Skill" %}
وصفة تشغيلية متخصصة تتفعّل عند سياق معيّن وتوجّه الوكيل عبر workflow محدد.
{% /tab %}
{% /tabs %}

## أمثلة سريعة

{% steps %}
{% step title="لا تحتاج Skill" hint="تفضيل بسيط" %}
أن تطلب من الوكيل أن يشرح بإيجاز أو يكتب commit messages بأسلوبك المفضل.
{% /step %}
{% step title="تحتاج Skill" %}
أن تطلب منه بناء README متعددة اللغات، أو اختبار webapp بـ Playwright، أو ضغط ملفات الذاكرة مع الحفاظ على الأكواد والروابط كما هي.
{% /step %}
{% step title="قد تحتاج Plugin لاحقًا" %}
إذا أصبح الـ workflow نفسه مطلوبًا لفريق كامل مع packaging واكتشاف وتثبيت.
{% /step %}
{% /steps %}

## الإشارة الذهبية

إذا كنت تشرح نفس الخطوات أكثر من مرة، أو تعتمد على checklist ذهنية ثابتة، أو تقول باستمرار "لا تنس أن..."، فغالبًا أنت أمام مرشح ممتاز لـ Skill.

{% checkpoint title="قبل الدرس التالي" %}
- لا تبن Skill لكل شيء.
- ابنِ Skill عندما توجد خبرة متكررة وقابلة للوصف والتفعيل.
- فرّق بين التفضيلات العامة والمهارات التشغيلية المتخصصة.
{% /checkpoint %}

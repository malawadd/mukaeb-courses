# Workflow تشغيلي وDecision Trees

العديد من Skills المتقدمة لا تكتفي بقائمة نصائح، بل تبني **منطق قرار**: إذا كانت المهمة من نوع معيّن فابدأ بهذه الخطوة، وإذا كان الخادم غير شغال فشغّل هذا السكربت، وإذا كانت الحالة static فاسلك طريقًا مختلفًا عن dynamic.

هذا هو الفارق بين "معرفة" و"تشغيل".

{% callout title="عندما يكثر التفرع، اكتب شجرة قرار" tone="note" %}
إذا كانت المهارة تحتوي على أكثر من مسار صحيح حسب الحالة، فلا تترك الوكيل يستنتجها من وصف عام. اكتب decision tree صريحًا.
{% /callout %}

## متى نحتاج decision tree؟

- عند وجود أكثر من نوع input أو بيئة تشغيل
- عند اختلاف الخطوات حسب حالة الخادم أو الملف أو النظام
- عند الحاجة للتمييز بين execution intent وguidance intent
- عند وجود fallbacks واضحة يجب اتباعها

## ماذا يجب أن يحتوي الـ workflow؟

1. خطوة preflight لجمع أقل سياق لازم.
2. قرار يحدد أي مسار سنسلك.
3. خطوات تنفيذ مرتبة.
4. خطوة تحقق أو sanity check.
5. مخرج نهائي واضح للمستخدم.

## مثال بنمط جيّد

```md
## Workflow
1. حدّد هل المستخدم يريد تنفيذًا مباشرًا أم شرحًا فقط.
2. تحقّق من الملف أو الخدمة أو المتغير المطلوب قبل أي تعديل.
3. إذا كانت الحالة A، افتح المرجع A أو شغّل السكربت A.
4. إذا كانت الحالة B، استخدم المسار البديل.
5. نفّذ التغيير على أضيق نطاق ممكن.
6. تحقّق من النتيجة وأبلغ المستخدم بما تغيّر.
```

## قاعدة "نفّذ أو اشرح"

بعض المهارات الحديثة تفرّق بوضوح بين:

- طلب يريد **تنفيذًا مباشرًا** على الملفات أو البيئة
- طلب يريد **إرشادًا فقط** أو template أو explanation

وجود هذا القرار داخل المهارة يمنع أن يكتفي الوكيل بإعطاء أمثلة عندما كان المطلوب تعديلًا فعليًا، أو بالعكس.

## أخطاء شائعة في كتابة workflow

- البدء بالتنفيذ قبل preflight
- تجاهل حالات الفشل أو fallback
- ترك التحقق النهائي ضمنيًا
- وصف المسار بعبارات فضفاضة مثل "افعل الشيء المناسب"

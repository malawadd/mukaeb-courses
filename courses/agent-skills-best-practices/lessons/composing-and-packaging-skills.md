# تركيب Skills وتغليفها

بعد أن تصبح المهارة جيدة محليًا، يظهر سؤال جديد: **كيف تتعايش مع مهارات أخرى؟ وكيف نوزعها على فريق أو أكثر من وكيل؟**

المستودعات المرجعية تعطي درسًا مهمًا هنا: المهارة الجيدة ليست معزولة عن بيئتها. أحيانًا تعيش داخل plugin، وأحيانًا داخل marketplace، وأحيانًا ضمن مستودع يوفّر نسخًا لوكلاء مختلفين أو ملفات packaging وmetadata.

## ما الذي يعنيه composition هنا؟

- أن تعرف متى يجب أن تفوز مهارتك على مهارة أخرى
- أن تحدد ما إذا كانت مهارتك stand-alone أم جزءًا من مجموعة
- أن توحّد naming conventions والوصف والهيكل عبر عدة Skills
- أن تمنع التعارض بين rules عامة وskills متخصصة

## مؤشرات أن الوقت حان للتغليف

{% steps %}
{% step title="تكرار الاستخدام بين عدة أشخاص" hint="فريق أو مجتمع" %}
إذا بدأت المهارة تنتقل بين المطورين أو المشاريع، فالتغليف يصبح استثمارًا منطقيًا.
{% /step %}
{% step title="الحاجة إلى metadata إضافية" %}
عندما تحتاج author/version/license أو تعريفات marketplace أو plugin manifest.
{% /step %}
{% step title="دعم أكثر من وكيل" %}
بعض المشاريع تضيف طبقات توافق أو مجلدات توزيع لأدوات متعددة بدل الاعتماد على مسار محلي واحد فقط.
{% /step %}
{% step title="وجود عدة Skills مترابطة" %}
إذا كانت لديك family كاملة من skills، فكّر في حزمة واحدة بمعايير موحدة.
{% /step %}
{% /steps %}

## ما الذي يجب توحيده داخل مجموعة Skills؟

- naming convention
- جودة الـ descriptions
- طريقة الإحالة إلى `references/` و`scripts/`
- شكل الأمثلة والحدود
- versioning وownership حيث أمكن

## تحذير مهم

{% accordion title="لا تبدأ بالتغليف قبل نضج السلوك" open=true %}
إذا كانت المهارة ما تزال تتغير أسبوعيًا، فابدأ محليًا أولًا. التغليف المبكر قد يجمّد تصميمًا لم يثبت بعد.
{% /accordion %}
